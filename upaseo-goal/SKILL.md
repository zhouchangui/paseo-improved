# Upaseo Goal Skill

**User's request:** $ARGUMENTS

本技能只做一件事：把用户的粗略描述转成一个短小、可执行、可验证的 `/goal`。在用户确认前，不得开始执行该 goal 对应的开发、调研、修复或发布工作。

## Step 0: 避障读取

如果当前项目根目录存在 `.paseo/learnings.jsonl`，先读取并提炼其中与“目标定义、范围控制、验证方式、工具误用”相关的避障规则。将这些规则作为 goal 草案的约束来源。若文件不存在则跳过。

## Workflow

1. **提炼意图**
   - 找出用户真正想要的最终结果，而不是复述任务动作。
   - 只保留两类信息：要完成什么目标、怎么验证完成。
   - 用户明确提到的硬约束可以融入目标或验证句子里，不单独展开成段落。
   - 如果缺少关键验证方式，优先给出合理默认验证；只有无法安全假设时才问 1 个问题。

2. **生成 goal 草案**
   - 使用“结果导向”的写法：`完成什么目标` + `怎么验证完成`。
   - Goal 必须能被另一个 Agent 独立执行，不依赖本轮对话中的隐含语境。
   - 不输出范围、非目标、约束、风险、假设、完成前审计等结构，除非用户明确要求。

3. **请求确认**
   - 向用户展示 goal 草案和关键假设。
   - 明确说明：回复“确认”/“开始”/“执行”后才会启动 `/goal`。
   - 用户提出修改时，更新草案并再次确认；不要启动执行。

4. **确认后启动**
   - 用户明确确认后，调用 `/goal <final goal>` 开始执行。
   - 如果当前运行环境提供 goal 工具且允许创建 goal，则使用工具创建同等内容的 goal。
   - 如果当前环境无法直接调用 slash command 或 goal 工具，则把完整 `/goal ...` 命令发给用户，并说明需要用户手动运行。

## Goal 草案模板

```text
目标：
【一句话说明最终要完成的结果。】

验证：
【一句话说明用什么命令、测试、日志、截图、页面状态或人工步骤判断完成。】
```

## Quality Bar

好的 goal 应该满足：

- **结果清楚**：读者知道最终要交付什么。
- **证据清楚**：知道用什么事实判断完成。
- **上下文足够**：另一个 Agent 不看本轮对话也能执行。
- **长度克制**：优先 2-6 行；复杂任务也先保持短句，避免写成项目章程。

## 输出格式

确认前只输出：

```text
我建议把 goal 写成：

<goal 草案>

回复“确认”后我会调用 /goal 开始执行；也可以直接告诉我要改哪一项。
```

确认后只执行 `/goal` 或等效 goal 工具调用，不要再追加新的设计讨论，除非用户修改了目标。
