# Upaseo Simplify (本地极致代码精简技能)

在代码合并、PR 提交前，**必须强制加载并调用此技能**，对所有的 `git diff` 更改进行全面脱水，确保交付的代码极度干净、凝练，符合 Senior 工程师的最高标准。

## Scope Boundary

- 精简范围默认限定为当前 `git diff`。只删除或收缩本次改动引入的冗余、过度抽象、废弃导入和多余注释。
- 不把“更喜欢另一种写法”当成精简理由；不重写未参与本次任务的旧代码。
- 如果发现旧代码存在明显坏味道但不影响本次交付，只在报告中记录，不顺手处理。

## 黄金法则

1. **Simplicity First (最简实现)**：
   - 拒绝一切非必要的抽象。如果一个类、函数或配置只在单一场景下使用，绝对不为它做通用包装。
   - 绝不夹带私货：只实现用户明确要求的特性，对于任何“未来可能有用”的代码，必须全部无情删去。
   - “如果本次 diff 中 200 行代码的功能可以用 50 行实现，请立即收缩它。”该原则只适用于当前改动范围，不授权重写无关旧代码。

2. **Surgical Changes (手术刀式改动)**：
   - **不乱改无关代码**：只修改实现本迭代目标所必须修改的行。禁止顺便整理相邻无关文件的格式、拼写或重构。
   - **打扫自己的战场**：仅清理由于“你自己的修改”而产生的废弃导入 (orphan imports)、未使用变量 (unused variables) 或废弃函数。
   - **尊重历史遗留**：不要随意删除历史已有的死代码（除非用户明确指示），防止产生潜在的破坏性影响。

## 前置避障读取

在执行精简规程前，检查当前项目根目录下是否存在 `.paseo/learnings.jsonl`。若存在，使用 `view_file` 读取并提炼避障规则。在精简审查过程中，若某段代码的存在正是为了遵守某条避障规则，则**禁止将其视为冗余代码删除**。若文件不存在则跳过。

## 精简执行规程 (Simplify Checklist)

在提交 PR 之前，加载此技能的 Agent 必须启动自检并完成以下工作：

1. **Diff 扫描 (Scan Diffs)**：
   - 运行 `git diff` 仔细观察每一处变更。
   - 提问自己：“这里有没有为了灵活性而设计的 speculative code (预测性代码)？”如果有，删掉它。

2. **精简死代码 (Trim Orphans)**：
   - 检查本次提交中是否有未使用的 `import`、声明了但未被调用的 `const/let/var`、已经无用的内部 helper。
   - 检查是否有由于重构而留下的多余注释。

3. **可读性审查 (Readability & Logic Shrinking)**：
   - 是否存在过长、嵌套过深的 `if-else` 或循环？能否用早期 return (early exit) 或更扁平的逻辑来平铺？
   - 确保风格与原项目保持 100% 绝对一致。即使你个人的品味更喜欢另外一种写法，也必须屈从于项目已有风格。

4. **确认最简形态 (Lock Minimum Shape)**：
   - 确保改动后的代码在不破坏原有功能的前提下，行数和复杂度达到了“Senior 工程师看后无懈可击的极简程度”。
   - 将精简结果以一句话记录在 commit message 或者是 PR 的自审日志中。
